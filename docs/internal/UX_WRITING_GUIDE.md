# Frapp UX Writing Guide

> Last updated: 2026-03-08

This guide standardizes microcopy across landing, dashboard, and mobile.

## 1) Tone

Frapp copy should be:

- Direct, clear, and operational
- Confident without marketing fluff
- Supportive under failure states

Avoid vague phrases like “Something happened” or “Please try again later” without context.

## 2) CTA Conventions

Use explicit action labels:

- ✅ “Get Started”
- ✅ “Create Event”
- ✅ “Adjust Points”
- ✅ “Invite Member”
- ❌ “Submit”
- ❌ “Continue” (when action is specific)

## 3) Error Copy Pattern

Use:

1. What failed
2. Why (if known)
3. What to do next

Template:

`<Action> couldn’t be completed. <Reason>. <Next step>.`

Example:

`Unable to load members. This view requires an authenticated chapter context. Sign in and retry.`

## 4) Empty State Pattern

Each empty state must include:

- A clear title
- A short explanation
- A next-step CTA when possible

Example:

`No events yet`  
`Create your first chapter event to unlock attendance and point automation.`  
`[Create first event]`

## 5) Status Labels

Status terms should match backend domain language:

- `ACTIVE`
- `PAID`
- `OPEN`
- `OVERDUE`
- `PENDING`
- `FAILED` (for transient action states)

Do not invent alternate terms for the same state on different platforms.

## 6) Trust Copy Rules

Billing, legal, and data-sensitive surfaces must:

- Use concrete language
- Avoid overpromising
- Explicitly indicate preview/demo data when shown

Examples:

- `Showing preview billing data`
- `Sign in to load live chapter subscription and invoice records`
