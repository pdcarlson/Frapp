## 2026-03-19 - [Supabase/PostgREST Filter Injection]
**Vulnerability:** Unsanitized user inputs in `.or()` filter strings
**Learning:** Supabase uses PostgREST which parses string values within `.or()` filters. When values are dynamically injected via template strings without escaping (e.g. `col.ilike.${value}`), an attacker can supply characters like `,` or `()` to break the query logic, leading to unexpected behaviors or data exposure. This project's `.or()` filters were constructing queries directly from user-provided search strings without quoting.
**Prevention:** All dynamic inputs used within Supabase/PostgREST `.or()` string filters must be escaped. Create and use an `escapeFilterValue` utility (e.g., `"${value.replace(/"/g, '""')}"`) to properly quote the values and double-escape internal double quotes according to PostgREST specification.
## 2024-05-24 - PostgREST Filter Injection via .or() Strings
**Vulnerability:** Found a PostgREST filter injection vulnerability in `SupabaseBackworkResourceRepository`. The repository was using unsanitized string interpolation inside a `.or()` query: `query.or(\`title.ilike.%${filters.search}%,course_number.ilike.%${filters.search}%\`)`. This allows users to input commas or parentheses to inject arbitrary PostgREST grammar elements into the query, potentially bypassing intended logic or causing errors.
**Learning:** When using raw strings for PostgREST filters in Supabase (like `.or()` or `.and()`), the values must be correctly escaped to handle special characters used by PostgREST's URL grammar (such as backslashes, double quotes, commas, and parentheses).
**Prevention:** Always use the `escapeFilterValue` utility function (from `apps/api/src/infrastructure/supabase/supabase.utils.ts`) to sanitize any user-provided string before injecting it into a raw string filter like `.or()`.
